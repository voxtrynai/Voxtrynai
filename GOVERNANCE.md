- Strategic direction
- Major feature decisions
- Release planning
- Community governance

**Members:**

## Decision Process

### Major Decisions (New Features, Architecture)

1. **Proposal** - Issue/discussion with detailed proposal
2. **Discussion** - Community feedback (7 days)
3. **Feedback** - Incorporate suggestions
4. **Decision** - BDFL makes final decision
5. **Implementation** - Execute decision

### Minor Decisions (Bug Fixes, Documentation)

- Handled through normal PR process
- Community review and feedback
- Merged when tests pass and reviewed

### Breaking Changes

- Require extensive discussion
- Community poll
- Major version bump
- Migration guide provided

---

## Conflicts & Disputes

### Resolution Process

1. **Private Discussion** - Direct communication between parties
2. **Mediation** - Neutral third party if needed
3. **Escalation** - Presented to Steering Committee
4. **Decision** - BDFL makes final decision
5. **Resolution** - Document and communicate outcome

## Project Maintenance

### Issue Triage

Issues are labeled and prioritized:
- **Critical** - Needs immediate attention
- **High** - Important for next release
- **Medium** - Should be done soon
- **Low** - Nice to have

### Pull Request Review

Standard review process:
1. Automated checks pass
2. Code review (≥1 approval)
3. Merge conflict resolution
4. Final approval
5. Merge to develop/main
